## Фаза 3.5 — Соответствие методу Бутаковой (на основе изучения материалов)

> Изучены материалы из папки Здоровье_Бутакова: скан книги 95 программ, зум-разборы диаграмм, концепция здоровья, 7 шагов метода, маршрутная карта.

### Что сейчас в инструменте НЕ соответствует методу:

1. **Нет уточнения состояния внутри системы** — метод содержит 95 конкретных программ.
   Например, для ЦНС: «бессонница», «антистресс для мужчин», «хроническая усталость», «депрессия», «головные боли» — это разные программы с разным набором продуктов.
   Сейчас Claude знает только «ЦНС в перегрузе» и подбирает всё подряд.

2. **Нет раздела «Коррекция питания»** — Шаг 4 метода Бутаковой. Каждый протокол включает рекомендации по питанию.

3. **Нет раздела «Дополнительные практики»** — Шаг 5: дыхание, ЛФК, детоксикация, психоэмоциональная коррекция.

4. **Пол клиента не используется для выбора программы** — поле есть, но протоколы для мужчин и женщин разные (особенно ЦНС, репродуктивная, иммунная).

5. **Возраст не используется** — для 60+ существуют отдельные программы (ЦНС 60+, память 60+).

6. **Нет концепции «Точка А → Точка Б»** — метод строится вокруг этого. В выводе должна быть цель через 90 дней.

### Что реализовать:

- [ ] **Уточняющий вопрос по главной системе** — после определения нагруженных систем добавить в форму поле «Основная жалоба» (выбор из списка по системе). Пример: ЦНС → [плохой сон / стресс / усталость / плохое настроение / головные боли]. Это позволит выбрать нужную из 95 программ.
- [x] **Использовать пол для протокола** — прописано в промпте: мужские/женские программы по системам
- [x] **Использовать возраст 60+** — программы для 60+ прописаны в промпте
- [x] **Добавить раздел «Коррекция питания»** — реализовано: питание по системам в SYSTEM_PROMPT + тег `<питание>` + карточка в UI
- [x] **Добавить раздел «Дополнительные практики»** — реализовано: практики по системам в SYSTEM_PROMPT + тег `<практики>` + карточка в UI
- [x] **Добавить «Точка А → Точка Б»** — реализовано через теги `<путь>`, `<зеркало>`, `<шаг_1>`

### Что нужно от тебя для полной реализации:
- Содержимое 95 программ из книги Бутаковой (конкретные продукты на каждую программу) — скан есть, но OCR распознал только оглавление. Нужно либо вручную перенести ключевые программы, либо найти цифровую версию книги.

## Фаза 3.17 — Оптимизация промпта: баланс безопасности и эффективности

> Цель: устранить два взаимосвязанных дефекта в SYSTEM_PROMPT. Первый — система избыточно применяет тег `<острая_фаза>` к управляемым хроническим состояниям (компенсированный диабет 2 типа, инфаркт/инсульт в анамнезе) — это медицински некорректно и коммерчески вредно: дистрибьютор теряет клиента которому можно помочь. Второй — выходной документ перегружен: некоторые блоки дублируют друг друга, а скрининг-вопросов слишком много и они звучат бюрократически.

### Проблема A — Логика острой фазы

**Текущее поведение (неверно):**
- `инсульт, инфаркт в анамнезе как "недавно"` → острая фаза → "ВСЕ БАДЫ — ПАУЗА"
- Сахарный диабет 2 типа (управляемый, сахар в норме) → острая фаза
- Тег `<острая_фаза>` включает заголовок "НУЖЕН ВРАЧ — И ТОЛЬКО ВРАЧ" + "ВСЕ БАДЫ — ПАУЗА"

**Правильное поведение:**
- Острая фаза = нескомпенсированное/нестабильное состояние прямо сейчас (глюкоза > 11, тромбоциты < 50, СРБ > 20, острый инфаркт/инсульт СЕЙЧАС)
- Хронический анамнез = контекст для протокола, не стоп-сигнал
- Управляемый диабет 2 типа = хроническая/восстановительная фаза → полный протокол с примечанием "врач в курсе"

**Изменения в `api/analyze.js`:**
- [ ] Убрать из триггеров острой фазы: `инсульт, инфаркт (даже в анамнезе как "недавно")`
- [ ] Уточнить: диабет → острая фаза ТОЛЬКО при глюкозе > 11 ммоль/л натощак; управляемый диабет = хроническая фаза
- [ ] Переписать шаблон блока `<острая_фаза>`: убрать заголовок "ВСЕ БАДЫ — ПАУЗА"; заменить на "Начинаем с ведома врача" — тон: сотрудничество, не запрет
- [ ] Добавить в 🟢 ХРОНИЧЕСКАЯ ФАЗА: примечание как вести хронические состояния (диабет, щитовидка, сердечно-сосудистые в анамнезе) — параллельно с медициной, с информированием лечащего врача

### Проблема B — Скрининг-вопросы

**Текущее поведение:** 4 базовых вопроса всегда + до 3-4 продуктовых = итого 5-7 вопросов. Выглядит как медицинская анкета, пугает клиента.

**Правильное поведение:** Скрининг = забота, не барьер. Максимум 3 вопроса суммарно, сформулированных как разговор.

**Изменения:**
- [ ] Базовые вопросы: сократить с 4 до 2 обязательных: (1) постоянные лекарства — что именно; (2) беременность/кормление
- [ ] Хронические болезни и онкология → перевести в контекстно-зависимые (только если анамнез неизвестен И протокол содержит рисковые продукты)
- [ ] Формулировки: переписать вопросы с анкетного языка на разговорный ("Принимаете что-нибудь постоянно?" вместо "Принимаете постоянные лекарства?")
- [ ] Суммарно не более 3 вопросов в `<скрининг>` в любом случае

### Проблема C — Избыточные блоки в выводе

**Текущее поведение:** блоки `<первопричина>`, `<цепочка>`, `<причина>` и `<готовое_сообщение>` все объясняют одно и то же — первопричину симптомов. Дистрибьютор видит один и тот же инсайт 4 раза в разных формулировках.

**Изменения:**
- [ ] `<логика>` — убрать из обязательных тегов; логика "почему эти продукты" должна встраиваться прямо в `<протокол>`, а не выноситься отдельно
- [ ] `<причина>` — сократить до 2 абзацев максимум (сейчас 3-4); это второе сообщение в цепочке, достаточно тезиса + образа
- [ ] Убедиться что `<первопричина>` (только для дистрибьютора) остаётся — это высокоценный аналитический инсайт
- [ ] Проверить что SYSTEM_PROMPT не требует генерировать `<цепочка>` И `<причина>` одновременно — это дублирование; `<цепочка>` = часть `<готовое_сообщение>` в свёрнутом виде

## Фаза 3.18 — База знаний: docs/knowledge/

> Цель: вынести доменные знания (продукты, протоколы, методология) из монолитного SYSTEM_PROMPT в отдельные структурированные файлы. Это устраняет "чёрные дыры" — когда информация теряется между сессиями — и делает базу знаний частью кодовой базы, а не только памятью ассистента.

### Проблема
~90% файла `api/analyze.js` (2721 строка) — это жёстко зашитые знания внутри строки SYSTEM_PROMPT. Добавить продукт = загрузить весь файл, рискуя сломать логику. Информация обсуждённая в разговорах с ассистентом не попадает в код автоматически — возникают "чёрные дыры".

### Архитектура (аналогия с памятью человека)
| Тип памяти | Где хранится | Что содержит |
|---|---|---|
| Долговременная | `docs/knowledge/` | Все продукты, протоколы, методология |
| Оперативная | Контекст сессии | Текущая задача, обсуждение |
| Процедурная | `api/analyze.js` (логика) | Как работает функция |

Файлы из `docs/knowledge/` деплоятся на Vercel вместе с кодом и читаются через `fs.readFileSync` при cold start. Лимит bundle 250 МБ — наши файлы займут <0.5%.
