## Фаза 3.5 — Соответствие методу Бутаковой (на основе изучения материалов)

> Изучены материалы из папки Здоровье_Бутакова: скан книги 95 программ, зум-разборы диаграмм, концепция здоровья, 7 шагов метода, маршрутная карта.

### Что сейчас в инструменте НЕ соответствует методу:

1. **Нет уточнения состояния внутри системы** — метод содержит 95 конкретных программ.
   Например, для ЦНС: «бессонница», «антистресс для мужчин», «хроническая усталость», «депрессия», «головные боли» — это разные программы с разным набором продуктов.
   Сейчас Claude знает только «ЦНС в перегрузе» и подбирает всё подряд.

2. **Нет раздела «Коррекция питания»** — Шаг 4 метода Бутаковой. Каждый протокол включает рекомендации по питанию.

3. **Нет раздела «Дополнительные практики»** — Шаг 5: дыхание, ЛФК, детоксикация, психоэмоциональная коррекция.

4. **Пол клиента не используется для выбора программы** — поле есть, но протоколы для мужчин и женщин разные (особенно ЦНС, репродуктивная, иммунная).

5. **Возраст не используется** — для 60+ существуют отдельные программы (ЦНС 60+, память 60+).

6. **Нет концепции «Точка А → Точка Б»** — метод строится вокруг этого. В выводе должна быть цель через 90 дней.

### Что реализовать:

- [x] **Уточняющий вопрос по главной системе** ✅ 2026-06-12 — поле «Главная цель клиента» в форме расшифровки: `<select>` с ~35 вариантами, сгруппированными по 12 системам. Передаётся в API как `mainGoal`, добавляется в контекст промпта — AI выбирает нужную программу из 95.
- [x] **Использовать пол для протокола** — прописано в промпте: мужские/женские программы по системам
- [x] **Использовать возраст 60+** — программы для 60+ прописаны в промпте

## Фаза 3.17 — Оптимизация промпта: баланс безопасности и эффективности

## Фаза 5 — Замыкание петли: Результат → Портфолио → Рост

**Стратегическая цель:** инструмент не только начинает путь клиента, но и фиксирует результат — превращая его в социальное доказательство, которое автоматически привлекает новых клиентов и создаёт почву для рекрутинга.

Полная цепочка: Трекер → Протокол → Сопровождение → **Результат → Кейс → Соцсети / Партнёрство**

### Фаза 5.1 — Фиксация результата клиента (Точка Б)

**Задача:** когда клиент завершил протокол (1-3 месяца), дистрибьютор вносит результат за 2-3 минуты. AI генерирует готовый пост.

**Механика:**
- В карточке клиента (вкладка «Клиенты») новая кнопка: **«Зафиксировать результат»**
- Форма из 6 полей (исследование подтвердило: это минимум для убедительного кейса):
  1. Главная жалоба до протокола (текст — 1-2 предложения от клиента)
  2. Продолжительность протокола (недель/месяцев)
  3. Один измеримый результат (вес, давление, сон, энергия — число)
  4. Субъективная оценка до/после по 3 параметрам: энергия, сон, самочувствие (шкала 1-5)
  5. Один конкретный момент ДО: «до протокола я...»
  6. Один конкретный момент ПОСЛЕ: «сейчас я...»
- AI генерирует 3 варианта поста (в словах клиента, не как реклама):
  - Telegram-формат (150-200 слов, список До→После)
  - VK-формат (300-350 слов, нарративная история)
  - Instagram-подпись (80 слов + хэштеги)
- Кейс сохраняется в карточке клиента и в раздел «Портфолио»

**Опросник для клиента (генерируется автоматически):**

На 30-й / 60-й день инструмент создаёт готовый опросник — персонализированный под конкретного клиента (вопросы по его ведущим системам из исходного трекера). Дистрибьютор копирует и отправляет в WhatsApp / Telegram. Клиент отвечает своими словами. Дистрибьютор вставляет ответы — AI строит кейс из слов самого клиента.

Структура (5 вопросов, не больше):
1. Оцени своё состояние по 10-балльной шкале: было ___ / сейчас ___ (общее самочувствие + энергия + ведущий симптом)
2. Что изменилось к лучшему? (своими словами)
3. Что пока осталось на том же уровне?
4. Что ещё хотела бы улучшить? ← естественное открытие следующего этапа работы
5. Как бы описала эти N недель подруге — одной фразой? ← готовая цитата для кейса

**Ключевые принципы (из исследования):**
- AI структурирует — но слова должны быть клиента. Без шаблонных «невероятно» и «потрясающе»
- Конкретная деталь («перестала просыпаться в 3 ночи») ценнее общего («стало лучше»)
- Вопрос 3 («что осталось») делает кейс честным и доверительным — не рекламным
- Фото не обязательно, но при наличии — удваивает вовлечённость

### Фаза 5.2 — Переход «Клиент → Партнёр»

**Задача:** после фиксации результата — AI автоматически предлагает дистрибьютору оптимальный момент и сценарий для разговора о партнёрстве.

**Механика:**
- После сохранения результата всплывает подсказка: «Клиент прошёл протокол и доволен результатом — это лучший момент для предложения партнёрства. Открыть сценарий?»
- Кнопка **«Подготовить разговор о партнёрстве»**
- AI строит персональный сценарий на основе:
  - Конкретного результата этого клиента (не шаблон, а его история)
  - Типа ведущей системы (человек с проблемой КМС получает другой акцент чем человек с ЦНС)
  - Психологического профиля (если был задан в расшифровке)
- Структура сценария (согласно исследованию — доказанная последовательность):
  1. Начать с их истории, не с возможности («Ирина, вы упомянули, что...»)
  2. Мягкий мост: «Вы думали о том, чтобы помочь другим получить то же самое?»
  3. Финансовый аргумент — только если клиент сам проявил интерес
  4. Низкий барьер входа как аргумент ($30 / стартовый набор)
- Готовое сообщение для WhatsApp / Telegram — скопировать и отправить

**Ключевые принципы (из исследования):**
- Лучший момент: 60-90 дней, когда клиент уже дал результат (не 30 дней — слишком рано)
- Триггер — не календарь, а событие: клиент сам поделился отзывом / спросил «где взять ещё»
- Никогда не начинать с денег. Начинать с миссии («помогать другим»)
- Barrier to entry Coral Club ($30) — сильный аргумент, использовать в конце

### Фаза 5.3 — Раздел «Мои кейсы» (Портфолио дистрибьютора)

**Задача:** место где собираются все зафиксированные результаты — для работы со своей аудиторией.

**Механика:**
- Новая вкладка или раздел внутри «Клиенты»: **«Мои кейсы»**
- Карточки кейсов с фильтрами по системе (ССС, ЦНС, КМС и т.д.)
- Для каждого кейса: кнопки «Скопировать для Telegram», «Скопировать для VK», «Скопировать для Instagram»
- Счётчик: «12 клиентов завершили протокол», «средний прогресс по энергии +2.4 балла»
- Опционально (Phase 2): экспорт в PDF для презентаций

### Фаза 5.4 — Автоматические чек-ины (день 30, день 60)

**Задача:** снизить отток клиентов в середине протокола — самая частая причина "не сработало".

**Механика:**
- При старте протокола дистрибьютор устанавливает дату начала
- На 30-й и 60-й день система напоминает: «Ирина на 30-м дне протокола — время написать»
- Кнопка «Написать чек-ин сообщение» — AI генерирует персональное сообщение для этого клиента
- Опция для клиента: ответить баллами 1-5 по энергии/сну/самочувствию
- Если ответил — AI обновляет прогресс и может скорректировать рекомендации

**Примечание:** это Phase 2 — реализуем после 5.1-5.3.

## Фаза 6 — Управление доступом и безопасность (2026-06-23)

> Исследование проведено 4 параллельными агентами по темам: invitation flows, SaaS security, Telegram Bot architecture, multi-tenant auth. Выводы синтезированы в единую архитектуру под наш стек (Vercel + Upstash Redis, plain HTML/JS, no framework).

### Проблема
- Партнёры не могли запросить доступ — не было формы, только страница тарифов с мёртвыми кнопками
- Владелец узнавал о запросах только из личных сообщений (нет системы уведомлений)
- Партнёры добавлялись через PARTNERS env var — требовал редеплой
- Нет защиты API от злоупотреблений (rate limiting только на клиенте)
- Один пароль на всех = риск шаринга и утечки

### Архитектура решения

**Флоу запроса доступа:**
1. Партнёр заполняет форму на pricing.html (имя + Telegram)
2. `api/request-access.js` → сохраняет в Redis (TTL 7 дней) → Telegram уведомление владельцу
3. Владелец нажимает ✅ в Telegram (одно касание)
4. `api/telegram-webhook.js` → генерирует уникальный пароль → сохраняет партнёра в Redis → отправляет пароль владельцу для пересылки
5. Владелец пересылает пароль партнёру в Telegram (~10 секунд работы)

**Redis-схема для партнёров:**
```
partners:passwords           HASH → { password: slug }   (O(1) поиск)
partner:{slug}:data          STRING JSON → { name, telegram, created, active }
partners:index               SET → [slug1, slug2, ...]
req:{requestId}              STRING JSON (TTL 7 дней) → { name, telegram, message, createdAt }
reqs:pending                 LIST → [requestId, ...]
ratelimit:analyze:{token}    STRING (TTL 1ч) → счётчик запросов
```

**Безопасность:**
- Rate limiting на `api/analyze.js`: 100 запросов / 1 час / токен (серверный, не localStorage)
- Canary token в системных промптах (обнаружение попыток извлечь промпт)
- Уникальный пароль на каждого партнёра (не общий)
- Деактивация партнёра мгновенно через Redis (без редеплоя)
- Новые партнёры в Redis (без редеплоя для добавления)

**Telegram Bot:**
- Новый бот через @BotFather
- Webhook: `api/telegram-webhook.js` (сырой Bot API без SDK, только fetch)
- env vars: `TG_BOT_TOKEN`, `TG_OWNER_CHAT_ID`, `TG_WEBHOOK_SECRET`
- Регистрация webhook после деплоя: одна curl-команда
